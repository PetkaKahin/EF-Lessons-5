# Задание 4. Индексы и EXPLAIN

Сравнение сделано на одинаковом наборе данных после повторного seed:

- `users`: 50 000 строк
- `products`: 20 000 строк
- `orders`: 100 000 строк
- `order_items`: 200 000 строк
- `payments`: 100 000 строк

Для сравнения "до индексов" временно удалялись только индексы из `migrations/007_add_indexes.sql`.
Индексы первичных ключей, `UNIQUE` и внешние ограничения PostgreSQL оставались, так как они создаются вместе со схемой.

Перед замерами был выполнен `ANALYZE`, в сессиях был отключен JIT: `SET jit=off`.

## Добавленные индексы

```sql
CREATE INDEX IF NOT EXISTS idx_orders_user_id ON orders (user_id);
CREATE INDEX IF NOT EXISTS idx_order_items_order_id ON order_items (order_id);
CREATE INDEX IF NOT EXISTS idx_order_items_product_id ON order_items (product_id);

CREATE INDEX IF NOT EXISTS idx_orders_created_at ON orders (created_at);
CREATE INDEX IF NOT EXISTS idx_payments_created_at ON payments (created_at);

CREATE INDEX IF NOT EXISTS idx_payments_status_created_at ON payments (status, created_at);
CREATE INDEX IF NOT EXISTS idx_orders_status_created_at ON orders (status, created_at);
```

## q1_top_users.sql

Запрос ищет пользователей с максимальной суммой оплаченных заказов за последние 30 дней.

До индексов:

```text
Seq Scan on orders
  Filter: ((status = 'paid') AND (created_at >= (now() - '30 days'::interval)))
  Rows Removed by Filter: 33132
Hash Join
HashAggregate
Execution Time: 112.369 ms
```

После индексов:

```text
Seq Scan on orders
  Filter: ((status = 'paid') AND (created_at >= (now() - '30 days'::interval)))
  Rows Removed by Filter: 33132
Hash Join
HashAggregate
Execution Time: 107.291 ms
```

Вывод: составной индекс `idx_orders_status_created_at` не был выбран. Условие `status = 'paid'` и дата за последние 30 дней возвращают примерно 66 тысяч строк из 100 тысяч, то есть слишком большую долю таблицы. Для такой выборки последовательное чтение дешевле, чем проход по индексу с большим количеством обращений к таблице.

## q2_products_sales.sql

Запрос считает продажи товаров за период `2026-04-28` - `2026-05-29`.

До индексов:

```text
Seq Scan on order_items
Seq Scan on orders
  Filter: ((created_at >= '2026-04-28') AND (created_at < '2026-05-29'))
  Rows Removed by Filter: 9672
Hash Join
HashAggregate
Execution Time: 139.507 ms
```

После индексов:

```text
Seq Scan on order_items
Seq Scan on orders
  Filter: ((created_at >= '2026-04-28') AND (created_at < '2026-05-29'))
  Rows Removed by Filter: 9672
Hash Join
HashAggregate
Execution Time: 155.207 ms
```

Вывод: индекс по `orders.created_at` не помог, потому что диапазон покрывает почти все заказы: найдено 90 328 строк из 100 000. В такой ситуации PostgreSQL снова выбирает `Seq Scan`. Индексы по FK тоже не дали выигрыша, потому что соединение идет массовое, а не точечное по одному заказу или товару.

## q3_orders_with_items.sql

Запрос получает последние 100 заказов вместе с позициями и товарами.

До индексов:

```text
Parallel Seq Scan on orders
Sort Key: orders.created_at DESC
Seq Scan on order_items
Hash Join
Execution Time: 34.369 ms
```

После индексов:

```text
Index Scan Backward using idx_orders_created_at on orders
Index Scan using idx_order_items_order_id on order_items
Index Scan using products_pkey on products
Incremental Sort
Execution Time: 1.679 ms
```

Вывод: это хороший пример полезного индекса. `idx_orders_created_at` позволяет быстро взять последние 100 заказов без сортировки всей таблицы `orders`. Затем `idx_order_items_order_id` позволяет найти позиции только для этих 100 заказов, а не читать все 200 000 строк `order_items`.

## q4_conversion.sql

```text
Seq Scan on orders

Aggregate

```

После индексов:


## q5_suspicious.sql

Запрос ищет пользователей, у которых было больше двух неуспешных платежей за 24 часа.

До индексов:

```text
Seq Scan on payments
  Filter: (status = 'failed')
  Rows Removed by Filter: 78146
Seq Scan on orders
Hash Join
WindowAgg
Execution Time: 63.837 ms
```

После индексов:

```text
Bitmap Index Scan on idx_payments_status_created_at
  Index Cond: (status = 'failed')
Bitmap Heap Scan on payments
Seq Scan on orders
Hash Join
WindowAgg
Execution Time: 46.348 ms
```

Вывод: составной индекс `idx_payments_status_created_at` помог отфильтровать платежи со статусом `failed`: вместо полного прохода по `payments` используется `Bitmap Index Scan`. Полностью сортировку для оконной функции индекс не убрал, потому что окно сортируется по `orders.user_id, payments.created_at`, а текущий индекс начинается со `status`.

## Общий вывод

Индексы полезны не сами по себе, а когда совпадают с реальным способом чтения данных.

Сильнее всего индекс помог в `q3`, где запросу нужны последние 100 заказов и их позиции. Там индекс резко сокращает количество прочитанных строк.

В `q1`, `q2` и `q4` фильтры выбирают большую часть таблицы, поэтому PostgreSQL оставляет `Seq Scan`. Это нормальное поведение: читать таблицу последовательно дешевле, чем идти по индексу ради десятков тысяч строк.

В `q5` индекс помог частично: платежи со статусом `failed` находятся быстрее, но дальнейшие соединение, сортировка и оконная функция все равно остаются основной частью работы.

Отдельно исправлена логическая несогласованность seed: теперь для заказа со статусом `new` не создается платеж со статусом `paid`.
